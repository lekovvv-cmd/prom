# AGENTS.md — инструкции для кодинг-агента

## Роль агента

Ты fullstack-разработчик. Твоя задача — реализовать MVP продукта «Витрина проектов ШПИУ» по файлам в этой папке.

Проект нужен как единое пространство для:
- руководства ШПИУ — видеть стратегические и приоритетные проекты, участников, перегруженные/недополучающие внимание направления;
- сотрудников и преподавателей — видеть текущие и прошедшие проекты, понимать цель, ответственного, рабочую группу и быстро откликаться;
- руководителей проектов — быстрее собирать команды и видеть заинтересованных людей.

## Главный принцип

Не усложняй MVP. Сначала реализуй рабочий end-to-end сценарий:

1. Администратор создаёт проект.
2. Сотрудник видит проект в витрине.
3. Сотрудник открывает страницу проекта.
4. Сотрудник отправляет отклик.
5. Администратор видит отклик и меняет его статус.

Frontend:

## Backend-архитектура

Backend обязательно разбить на три основных слоя/модуля:

```txt
backend/
  app/
    api/
    core/
    modules/
```

### `api`

`api` — только HTTP-слой FastAPI.

В `api` должны лежать:
- роутеры;
- зависимости FastAPI;
- обработка path/query/body параметров;
- подключение роутов к приложению.

В `api` запрещено размещать бизнес-логику.

Роуты должны вызывать сервисы/use cases из `modules`.

### `core`

`core` — общий независимый слой.

В `core` должны лежать:
- базовые настройки;
- DTO / схемы;
- перечисления;
- общие исключения;
- базовая SQLAlchemy-конфигурация;
- общие интерфейсы, если нужны;
- общие утилиты без привязки к конкретному модулю.

Важно:
- `core` не должен импортировать ничего из `modules`;
- `core` не должен зависеть от конкретной бизнес-логики;
- `core` не должен запускать приложение;
- `core` не должен инициализировать конкретные фичи.

### `modules`

`modules` — бизнес-модули продукта.

В `modules` должны лежать:
- модели SQLAlchemy;
- репозитории;
- сервисы/use cases;
- модульные схемы, если они специфичны для конкретной области;
- логика создания, редактирования, архивации проектов;
- логика откликов;
- логика пользователей и ролей;
- логика статистики.

`modules` может импортировать из `core`.

`modules` может вызываться из `api`.

## Frontend-архитектура

Frontend делать строго по Feature-Sliced Design.

Базовая структура:

```txt
frontend/src/
  app/
  pages/
  widgets/
  features/
  entities/
  shared/
```

### Слои FSD

`app`
- провайдеры;
- роутинг;
- глобальные стили;
- инициализация приложения.

`pages`
- страницы приложения;
- сборка виджетов и фичей в экраны.

`widgets`
- крупные блоки интерфейса: ProjectList, AdminProjectsTable, ResponsesTable, Header, Layout.

`features`
- пользовательские действия: createProject, editProject, archiveProject, submitProjectResponse, updateResponseStatus, filterProjects.

`entities`
- бизнес-сущности: project, user, projectResponse, tag.

`shared`
- UI-kit;
- API client;
- lib/helpers;
- config;
- shared types.

## Ограничения MVP

Сейчас в системе есть только стратегические проекты, но архитектура должна позволять добавить другие типы позже.

Даты проекта опциональные.

Авторизация в MVP может быть упрощённой:
- пользователь вводит email;
- email должен быть на домене `@utmn.ru`;
- полноценную отправку кода можно заменить dev-flow или mock-flow, но API должен предусматривать:
  - `POST /api/auth/request-code`
  - `POST /api/auth/verify-code`
  - `GET /api/me`

Нельзя делать сложную корпоративную SSO-интеграцию в MVP.

## Качество кода

- Используй типизацию.
- Не пиши бизнес-логику в роутерах.
- Не смешивай backend и frontend.
- Не дублируй enum-значения руками в разных местах без необходимости.
- Добавь README с инструкциями запуска.
- Добавь `.env.example`.
- Добавь alembic migrations.
- Добавь seed-данные для демо.
- Предусмотри CORS для локального frontend.
- Возвращай понятные ошибки API.

## UX MVP

Интерфейс должен быть простым и понятным:
- витрина проектов карточками;
- фильтры по статусу, типу, поиску;
- сортировка по дате создания / приоритету;
- страница проекта;
- форма отклика;
- админ-таблица проектов;
- админ-таблица откликов;
- статусы откликов с возможностью изменения.
