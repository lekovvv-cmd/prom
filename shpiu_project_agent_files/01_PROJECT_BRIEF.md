# 01_PROJECT_BRIEF.md — бриф продукта

## Название

Витрина проектов ШПИУ

## Суть продукта

Продукт — это внутренняя витрина стратегических и приоритетных проектов ШПИУ, где сотрудники и преподаватели могут видеть проекты, понимать их цель, ответственных, рабочие группы и откликаться на участие.

Для администрации это инструмент прозрачности проектной деятельности и сбора откликов.

## Бизнес-ценность

### Для руководства ШПИУ

Продукт даст руководству единое пространство, где можно видеть:
- все стратегические и приоритетные проекты;
- ответственных;
- участников;
- статусы;
- количество откликов;
- какие направления перегружены;
- какие направления не получают внимания.

### Для сотрудников и преподавателей

Продукт станет понятной витриной прошедших и текущих проектов.

Он снижает порог входа в проектную деятельность, потому что сотрудник получает единый источник информации:
- какие проекты есть;
- зачем они нужны;
- кто отвечает;
- какая рабочая группа уже собрана;
- какие компетенции нужны;
- как можно откликнуться.

### Для руководителей проектов и направлений

Руководители смогут:
- быстрее собирать команды;
- видеть заинтересованных сотрудников;
- понимать, кто хочет участвовать;
- в перспективе видеть, кто в каких задачах участвовал и какие компетенции проявлял.

В MVP компетенции можно собирать через текстовое поле в отклике. Полноценную аналитику компетенций не делать, но заложить структуру для будущего развития.

## Проверяемая гипотеза

Продукт должен помочь проверить ключевую гипотезу:

> Проблема низкой вовлечённости сотрудников в проекты связана с плохой коммуникацией и отсутствием прозрачной витрины проектов.

Если после запуска прозрачной витрины откликов станет больше, значит продукт закрывает реальную боль.

Если откликов не станет больше, руководство получит данные, что проблема глубже: не только в информировании, но и в мотивации сотрудников.

## MVP

В MVP должны быть:

### Публичная часть / часть сотрудника

- список проектов карточками;
- фильтр по типу проекта;
- фильтр по статусу;
- поиск по названию;
- сортировка по дате создания / приоритету;
- страница проекта;
- форма отклика на проект.

### Административная часть

- таблица проектов;
- создание проекта;
- редактирование проекта;
- архивирование проекта;
- список откликов;
- фильтр откликов по проекту;
- изменение статуса отклика;
- базовая статистика.

## Пользователи и роли

### Сотрудник / преподаватель

Основной пользователь витрины.

Может:
- смотреть список проектов;
- открывать карточку проекта;
- читать описание, цель, ответственного, рабочую группу;
- откликаться на проект;
- указывать комментарий, интерес, компетенции или контактные данные.

### Администратор

На MVP это может быть директор, заместитель директора или помощник директора.

Может:
- создавать проекты;
- редактировать проекты;
- скрывать / архивировать проекты;
- смотреть список откликов;
- менять статус отклика;
- видеть базовую статистику.

### Руководитель проекта / направления

На первом этапе может быть объединён с администратором.

В перспективе должен иметь права:
- редактировать свои проекты;
- смотреть отклики только по своим проектам;
- управлять рабочей группой проекта.

## Главный пользовательский сценарий

1. Администратор создаёт проект.
2. Проект появляется в витрине.
3. Сотрудник открывает витрину.
4. Сотрудник фильтрует или ищет проект.
5. Сотрудник открывает страницу проекта.
6. Сотрудник нажимает «Откликнуться».
7. Сотрудник заполняет форму.
8. Отклик сохраняется.
9. Администратор видит отклик в панели.
10. Администратор меняет статус отклика.

## Не делать в MVP

- сложную аналитику компетенций;
- полноценную SSO-интеграцию;
- сложные workflow согласования;
- уведомления в почту;
- личные кабинеты с историей участия;
- сложный дизайн-конструктор проектов;
- матрицу загрузки сотрудников.

Эти функции можно заложить архитектурно, но не реализовывать в первой версии.
